Privacy Policy
Effective date: July 2026 · vaultlix.com
The short version: We collect nothing about you. We store nothing about your conversations. When your session ends, everything is permanently gone.

What we collect

Nothing that identifies you. Vaultlix does not require an account, email address, phone number, or any personal information to use.

What exists on our servers is:

  • The encrypted content of messages, held in server memory (never written to disk) so it can be delivered to the other person — not "in transit" only, but for the vault's lifetime; see "How long things last" below
  • The display name each person chooses when creating or joining a vault
  • If notifications are enabled, a Web Push subscription (an endpoint address tied to your device, provided by Apple's or Google's push service) so a message or call can wake the app
  • If a vault is created with a password, a one-way cryptographic hash of that password (scrypt) — never the password itself
  • Standard server access logs generated by Railway (our hosting provider) which may include IP addresses and request timestamps
  • Anonymous, aggregate usage counts (for example, how many vaults are created and whether they're temporary or permanent) — never linked to a specific vault, code, or IP address, kept indefinitely as running totals only, and used only to understand which features are actually useful

A session token is also kept on your device — in your browser's own localStorage, so the app can restore your open vaults if you reload or reopen it, and in IndexedDB if notifications are enabled, so a push notification can be matched to the right vault without a browser tab needing to be open. Neither is ever sent to or stored on our servers.

How messages work

All messages are end-to-end encrypted using AES-256-GCM with keys generated in your browser. The server never sees your message content — only encrypted ciphertext that it cannot read.

There is no traditional database and no ongoing backups — server-side data lives only in memory during normal operation. The one exception: in the brief moments during a planned server restart (for example, a deployment), the server writes a temporary snapshot of that same in-memory data to disk purely so open vaults aren't wiped out, then reloads and deletes that snapshot the moment it's back online. This snapshot holds nothing beyond what the server already had in memory — the same encrypted message content, not new plaintext — and it does not help against an unexpected crash, only a graceful restart.

How long things last

A vault and everything in it (messages, member names, subscriptions) is deleted immediately if either person uses "Close & erase." Otherwise, a temporary vault is automatically and permanently deleted after 24 hours of inactivity, measured from the last activity in that vault, not from when it was created. Simply closing your browser tab does not immediately delete a vault; the idle timer above is what eventually does. A permanent vault does not have an inactivity timer at all — it stays available indefinitely until someone uses "Close & erase" or revokes it from within the app, at which point it and everything in it are deleted the same way.

Disappearing-message timers, when enabled, delete an individual message from the server as soon as its timer expires, independent of the vault's own lifetime.

File attachments

Files are encrypted in your browser before upload, transmitted as encrypted data, and delivered directly to the recipient. Files are not stored on our servers beyond the same vault lifetime described above.

Voice and video calls

Calls are set up over an encrypted signaling channel, and the connection itself is encrypted in transit (DTLS-SRTP, standard for WebRTC calls). To protect both callers' IP addresses from being exposed to each other, call audio — and video too, if either person turns their camera on — is always relayed through a TURN server operated by Cloudflare, rather than connecting the two devices directly. This means a call is not end-to-end encrypted the same way text messages are — Cloudflare's relay is technically an intermediary point in the connection, though it is not a party we share data with beyond what's needed to relay the call, and no call audio or video is recorded or stored by Vaultlix.

Cookies and tracking

We do not use cookies, analytics trackers, advertising pixels, or any third-party tracking tools. We do not use Google Analytics or any equivalent service.

Third-party services

Vaultlix is hosted on Railway (railway.app). Railway may collect standard infrastructure logs including IP addresses. Please review Railway's privacy policy for details. We use Google Fonts for typography — fonts are loaded from Google's servers. Voice and video calls are relayed through Cloudflare's TURN service. Push notifications are delivered through Apple's and Google's respective push notification services, which see that a notification was sent to your device but not its content.

Children's privacy

Vaultlix is not intended for use by anyone under the age of 18. We do not knowingly collect information from minors.

Legal requests and abuse reports

If we receive a valid legal request — a court order or a request from law enforcement made under applicable Indian law — we will review it and respond as required by law. What we can actually provide is limited by how Vaultlix is built: we can only disclose what exists in server memory at the moment the request is received (the encrypted content of an active vault, the display names in it, and whatever standard access logs Railway retains) — we cannot decrypt message content ourselves, and we cannot produce anything for a vault that has already been closed or has already expired, because nothing remains anywhere once that happens.

To report abuse, suspected illegal content, or a violation of our Terms of Service, contact legal@vaultlix.com. We may act on a credible report by disabling or deleting the vault in question. We do not proactively monitor message content — because it's end-to-end encrypted, we couldn't even if we wanted to — so we only become aware of what a specific vault contains if it's reported to us, or compelled by valid legal process while that vault still exists.
